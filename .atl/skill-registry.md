## Compact Rules

Pre-digested rules to inject into sub-agent prompts as `## Project Standards (auto-resolved)`. Match by file context (extensions / paths) AND task context (PR creation, review, testing, etc.).

### General — applies to all phases

- Use pnpm for every install / script command. Never use npm or yarn.
- Conventional commits only (`feat:`, `fix:`, `refactor:`, `chore:`, `docs:`, `test:`). No AI co-author tags.
- Prefer editing existing files over creating new ones.
- Default to short, direct responses in Rioplatense Spanish (voseo) when talking to the user; English only if user switches.
- Use `bat`, `rg`, `fd`, `sd`, `eza` instead of `cat`, `grep`, `find`, `sed`, `ls`.
- Never run `pnpm build` or `next build` after changes unless explicitly asked.

### TypeScript / Next.js — `**/*.ts`, `**/*.tsx`, `app/**`, `lib/**`

- Use App Router conventions: `app/<route>/page.tsx`, `app/<route>/route.ts` for handlers, `app/<route>/layout.tsx`.
- Server Components by default; mark client components with `"use client"` only when state, effects, or browser APIs are required.
- Strict TypeScript — no `any`, no implicit `any`. Prefer `unknown` + narrowing.
- Co-locate component-specific tests next to the component (`Foo.tsx` + `Foo.test.tsx`).
- Use absolute imports via `@/` alias (configured in `tsconfig.json`).
- Server-only secrets must NEVER appear in client bundles — keep them behind `process.env.X` in server contexts only.

### Supabase — `lib/supabase/**`, anything touching `@supabase/*`

- Use `@supabase/ssr` for Next.js integration (NOT the deprecated `@supabase/auth-helpers-nextjs`).
- Two client factories: `createServerClient` (Server Components / Route Handlers / middleware) and `createBrowserClient` (Client Components).
- Every table that holds user data MUST have RLS enabled with policies — never disable RLS as a shortcut.
- Migrations live in `supabase/migrations/*.sql` and are applied via `supabase db push` or the MCP `apply_migration` tool.
- Generate types with `supabase gen types typescript` → `lib/database.types.ts`. Import `Database` type in clients.

### Database design — schema changes, migrations

- Use UUID primary keys (`uuid` with `default gen_random_uuid()`).
- Every table gets `created_at` and `updated_at` timestamps (`timestamptz default now()`).
- Foreign keys ALWAYS get an explicit index — Postgres does not create one automatically.
- Add `ON DELETE` behavior explicitly (`cascade`, `set null`, `restrict`) — never leave it implicit.
- Soft delete only when there's a clear undo/audit requirement; otherwise hard delete.

### UI / Styling — `app/**`, `components/**`, `*.tsx`

- Tailwind utility classes only; no inline styles unless dynamic values require it.
- Animations via Framer Motion for component-level motion; CSS transitions for hover/focus micro-interactions.
- Accessibility is non-negotiable: every interactive element needs a label, focus state, and keyboard reachability.
- "girlypop" palette will be defined in `tailwind.config.ts` — reference design tokens, never hardcode colors.

### Testing — `**/*.test.ts`, `**/*.test.tsx`, `tests/**`, `e2e/**`

- Vitest for unit/component, Playwright for E2E.
- Test the behavior, not the implementation. No snapshot tests for component HTML.
- E2E tests run against a `pnpm dev` server with a seeded local Supabase (or against a Vercel preview URL in CI).
- Use `@testing-library/react` queries in priority order: `getByRole`, `getByLabelText`, `getByText`, then `getByTestId` as last resort.

### Git / PR — when creating commits, branches, or PRs

- Branches: `feat/<slug>`, `fix/<slug>`, `chore/<slug>`, `docs/<slug>`.
- Keep PRs under 400 changed lines. If larger, use `/chained-pr` to split.
- PR description: Summary (1-3 bullets) + Test plan (checklist).
- Never force-push to `main`. Squash-merge PRs into `main`.

### Security review — before any auth, RLS, or secret-handling change

- Run `/security-review` on auth flows, RLS policies, and anything that touches `process.env.SUPABASE_SERVICE_ROLE_KEY`.
- Never expose service role keys to the client. Use them ONLY in server contexts (Route Handlers, Server Actions).
